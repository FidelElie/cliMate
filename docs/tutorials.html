Conventions

CliMate's declarative nature leads to numerous code snippets and to make them more clear, this documentation also shows you the states of files before and after the changes you make. Consequently, to minimise the size of each of the code blocks, two methods of condensing is used:

Condensing Code Structures

After a code block has been declared previously, usually in subsequent code blocks it will be condensed with ellipses unless it is the focus of our coding.


{
    "general":{...},
    "commands": {
        "new-command-1": {...}
    }
}

Skipping File Content

Used in python files if a large amount of previous code in a file has remained unused in our coding. It will be omitted from the before and after code progression through the use of a hashtag followed by a ellipsis. 


# ... Means Code Before

def new_function(argument_1, argument_2):
    pass

A Closer Look

As you may have already noticed, CliMate revolves around commands. Each of which holds a name, description, target and optional arguments. Instead of just passing your desired data directly to the file object everything is organised by these commands. The command call is defined by its key; in the case of the example above this is say-hello. The "name" and "description" keys are used in creating the Menu and Help structures and should be brief but concise.

The "target" key is where the magic happens and deserves its own section entirely to see what it is capable of here. However, for now its purpose is to tell the application where it should send the arguments too and call the given object. This can be in a function as in the example above or other first class objects like methods and classes. Above we define the target type as a function. Arguments are defined in a similar matter to the command itself. They possess "name" and "description" keys for the same purposes. As well as keys for defining the desired data type of the input and default values if desired.

We define all this commands but to start CliMate running we use python to load the Cli File and start the parsing process. This is done with the line below in the cli.py file:


climate.parse_args()

After this is called CliMate takes in the commands, parses and converts arguments, resolves the target and calls it with all of its required arguments.

How Targets Work

In CliMate, targets pass all argument information to the desired python object through the use of a sys.modules call. So any modules that are loaded in at runtime can be called upon. The chosen command is resolved from the Cli file, then the target is resolved from the dictionary returned from sys.modules. If the "module-name" for a target is omitted however, the object is searched for in the current running file (i.e sys.modules.get("__main__"))

We have currently only explored one of the types of targets that we can point CliMate to. Luckily for us they all work the same way. CliMate commands can also be pointed at python classes and methods and which one you use is based on the project you are attempting to write or the pre-existing structure of your project. All arguments are passed to each object accordingly. The display below shows the key required to utilise each of the target types and how CliMate passes arguments to it.

Target Tips

The module name defines where Climate looks for a package but it doesn't tell it what object to call. If you leave out the name of your object for example the "function-name", CliMate will take the name of your command for example "new-app" in this case and convert the hyphen to an underscore (new_app) and then search for an object of this name. Note that this will return an error if you are looking for methods. So as a rule of thumb always add the name of your target object to be consistent and less prone to error.

    def new_command(cli_path, arg_amount, target_type):
        utilities.check_cli_path(cli_path):
        command_contents = data.COMMAND_CONTENT.copy()
        target = targets[target_type] if target_type is not None else {}
        command_contents["target"] = target

        if arg_amount != 0:
        arguments = {}
        for i in range(arg_amount):
            arguments[f"new-arguments-{i}"] = data.ARGUMENT_CONTENT.copy()

        cli_commands = cli_present["commands"]
        command_amount = len(cli_commands.keys())
        cli_commands[f"new-command-{command_amount + 1}"] = command_contents
        cli_present["commands"] = cli_commands
        utilities.write_json(cli_path, cli_present)

Advanced

The astute among you may have noticed that there are links between some of the command arguments on offer. For example the "command-name" is based on what cli file is loaded and the "argument-name" is based on whatever command was chosen from the command name. For conventionally inputting commands this is just a very simple coincidence but for the menu this introduces some interesting opportunities. As each arguments' data is loaded upon the users' choice between being prompted for each argument. We can use a "map" key in these arguments to utilise the data previously loaded from the previous argument. So instead of the user having to enter the command they want to remove or add an argument to manually, they are presented with a list of the commands present in the cli file.

The "map" key contains a string of comma separated commands to return the data from a previous argument and extract the data from this argument. Lets add the "map" to the general argument "command-name" and we can explain it further.

The mapping is similar to the previous, except when we return the dictionary given by "cli-path" and then we have to navigate deeper into the hierarchy of the dictionary. To achieve this we pass keys separated by a | inside the curly brackets. CliMate will navigate the tree and substitute values for any argument keys it finds. For example, in this case the key we chose previously for the "command-name" is substituted into this navigation and its arguments are returned for the user to choose from. You can test the "remove-argument" command on your cli-test.json file and see how it works. This is much easier than having to input these manually so don't forget that this is an option if you find any intrinsic links between arguments in your commands.

Mapping Reference

"read {}"

This command takes an argument that corresponds to a file path. It will attempt to read the data from the file object. It will return a dictionary if the file contains the JSON extension and a list for anything else.

"get |keys|line| {}"

This command can take two arguments inside the placeholder ({}) depending on what modifier you choose. The "keys" modifier is used for navigating returned dictionaries and the line will return a line for the file in question.

Settings Reference

Currently, these are all the settings that you can change in CliMate. These should be able to cover any of your current needs for the package.

"use_menu"

Boolean Type

Determines whether CliMate applications has a menu. This is set to True by default.

"docs_path"

String Type

Sets path to application documentation, either a url or file path.

"app_name"

String Type

Sets desired application name, to be displayed in main menu.

"menu_help"

Boolean Type

Determines whether "Help" option is added to standard menu (When no menu is present in cli.json file). Will have no affect if "use_menu" settings is set to False. Set to True by default.

"menu_exit"

Boolean Type

Determines whether "Exit" option is added to standard menu (When no menu is present in cli.json file). Will have no affect if "use_menu settings is set to False. Set to True by default.

"pass_by_dict"

Boolean Type

Determines if instead of passing arguments through list unpacking function(*arguments), dictionary unpacking function(**arguments) is used. Set to False by Default.

"help_message_menu"

String Type

Sets message that is display on help sub menu.

"exit_upon_command"

Boolean Type

Sets whether the application will exit after completing command or it will loop back to where the user was in the menu hierarchy. This setting will only take affect if "menu" field is present in cli as well as the "use_menu" settings being set to True.

When creating your own projects you might find yourself using the setuptools library to add entry_points for your scripts that utilise CliMate. if this is the case you will want to add the path command to your CliMate application so the help function can display it. Below shows what this would look like in your CliMate clone application with the calls that are used for the CliMate package itself.

{

        "arguments": {...},
        "menu": {...}
    },
    "commands": {...}
}
                        

{
    "general": {
        "calls": ["clm", "climate"],
        "arguments": {...},
        "menu": {...}
    },
    "commands": {...}

Prompt

If you're in the market for a command line application solution such as this one, you will probably want to make custom menus elsewhere in your application. The amazing PyInquirer library powers the menus in CliMate with a few modifications. The most notable is the modifications to the default prompt command that allows KeyboardInterrupt signals to trigger if the user wants to exit the application. To utilise it, write the line below into your python file to import the function.


from climate import prompt

A list containing standard PyInquirer questions or just one question can be passed to this function for you're menu navigation. Basic snippets are shown below for this but for more information on how to use this library to its fullest potential check their Github here, they did some amazing work.

One Question Prompts


from climate import prompt

question = {
    "name": "value", # key in returned dict to get value
    "type": "input", # the type of input used
    "message": "Choose A Name" # the message to prompt the user with
},

returned_dictionary = prompt(question)
returned_value = returned_dictionary["value"]

Multi Question Prompts


from climate import prompt

questions = [
    {
        "name": "firstName",
        "type": "input",
        "message": "Enter First Name"
    },
    {
        "name": "secondName",
        "type": "input",
        "message": "Enter Second Name"
    }
]
# returned dictionary contains the keys that "name" values in questions
returned_dictionary = prompt(questions)
first_name = returned_dictionary["firstName"]
second_name = returned_dictionary["secondName"]
print("Hello {} {}".format(first_name, second_name))
